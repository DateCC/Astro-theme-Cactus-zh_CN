在 ESXi 环境下扩容 x86 OpenWrt 固件时，由于固件普遍采用 SquashFS 格式，直接放大第二分区（`sda2`）常会因内核锁死或缺少 `resize2fs` 工具而失败。

本文介绍一种 成功率 100% 的行业标准做法：保持原系统盘不动，利用 ESXi 扩容的未分配空间，动态切出一个独立的第三分区（`sda3`）来接管 `overlay`（系统软件包存储区），实现无损扩容。

## 📥 第一步：ESXi 后台调整虚拟硬盘大小

1. 登录 ESXi 后台，关闭 你的 OpenWrt 虚拟机。
2. 点击 “编辑” 设置，找到 硬盘 1（装有固件的盘）。
3. 将容量修改为你想要的最终大小（例如：从几百兆修改为 `2GB` 或 `5GB`）。
4. 保存设置，将 OpenWrt 虚拟机 开机。

## 🛠️ 第二步：登录终端进行动态物理分区

等待软路由开机网络恢复后，通过 SSH 或 ESXi 控制台进入 OpenWrt 终端，依次复制运行以下命令：

```
# 1. 确保安装了必要的分区与格式化工具
opkg update && opkg install parted mkfs.ext4

# 2. 【万能动态行】自动抓取 sda2 的物理终点，一秒不差地作为 sda3 的起点切出新分区
# 此命令可自动适配 100M、500M 或 1G 等任意体积的固件，绝不报分区重叠错误
parted -s /dev/sda mkpart primary ext4 $(parted /dev/sda print | grep "2 " | awk '{print $3}') 100%

# 3. 强行刷新内核分区表，让系统认出 sda3
partprobe /dev/sda

# 4. 格式化这个全新、干净的 sda3 数据盘
mkfs.ext4 -F /dev/sda3
```

## 🚚 第三步：配置与数据无损迁移

继续在终端执行以下命令。这一步会将你当前系统里已有的配置、密码和插件无损复制到新分区中，确保重启后网络不失联：

```
# 1. 创建临时目录，并将刚刚格式化好的 sda3 挂载上去
mkdir -p /tmp/newsda3 && mount /dev/sda3 /tmp/newsda3

# 2. 将当前系统 /overlay 的所有底层文件无损复制到新分区
cp -a /overlay/. /tmp/newsda3/

# 3. 复制完成后，卸载临时挂载
umount /tmp/newsda3
```

## 🔒 第四步：LuCI 网页端强行挂载（底层避坑）

> 💡 为什么不在终端用 fstab 挂载？部分定制固件（如 ImmortalWrt）带有特殊的底层机制，开机会自动创建虚拟的 `loop0` 盘强占 `/overlay`。为了防止终端配置被系统策略无视，最稳妥、优先级最高的方法是直接去网页端（LuCI）图形界面上保险。

1. 登录 OpenWrt 网页后台，进入 系统 (System) -> 挂载点 (Mount Points)。
2. 滚动到页面下方 “挂载点 (Mount Points)” 区域，点击 “添加 (Add)”（若列表中已有 `sda3` 则直接点击“编辑”）。
3. 严格按照以下参数进行调整：

   * 启用 (Enabled)：打勾 ✅
   * 设备 (Device)：在下拉菜单中精准选择 `/dev/sda3`（通常会带有 UUID 及新扩容的大小提示）。
   * 挂载点 (Mount Point)：在下拉菜单中精准选择 `作为外部 overlay 使用 (custom)`（若无此选项，则手动输入 `/overlay`）。
4. 点击右下角的 “保存并应用 (Save & Apply)”。

## 🚀 第五步：重启与验证

回到终端，输入最后通牒让其生效：

```
reboot
```

### 🏁 验证结果

软路由重启完成后，再次进入终端输入：

```
df -h
```

完美扩容成功的标志：你会看到挂载在 `/overlay` 和根目录 `/` 的设备变成了 `/dev/sda3`，且可用空间（Available）变成了你此前在 ESXi 中调整的大容量（如 1.6G+）。

至此，扩容彻底完成，你可以开始无限制地下载 Docker 镜像或各类大体积插件了！
